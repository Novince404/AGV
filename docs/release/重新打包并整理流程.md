# 重新打包并整理流程

> 状态：待审核。本文档先固化“重新打包并整理”的执行标准，审核通过后再按此流程移动旧版本并重新封装。

## 触发口令

当用户说“重新打包并整理”时，默认执行本文档流程。

执行范围包括：

- 归档已有封装成品。
- 重新构建综合版 / 个人端可用包。
- 重新构建企业端独立客户端包。
- 使用中文目录名整理最新版本，方便直接查看和交付。
- 验证启动脚本、核心 exe、demo 资产和说明文档是否存在。

## 目标目录结构

整理完成后，`dist` 目录建议保持如下结构：

```text
dist/
  旧版本/
    归档_YYYYMMDD_HHMM/
      AGV_Dispatch_Package/
      AGV_Dispatch_Package_beta/
      AGV_Dispatch_Package_v2/
      AGV_Enterprise_Client_v1/
  最新版本/
    AGV调度系统_综合版_YYYYMMDD/
    AGV企业端独立客户端_YYYYMMDD/
```

说明：

- `旧版本` 用于保存本次打包前已经存在的用户可运行封装目录。
- `最新版本` 用于保存本次重新打包生成的最新成品。
- `YYYYMMDD_HHMM` 用当前执行时间，避免覆盖历史归档。
- 旧版本目录内部优先保留原英文目录名，方便和脚本输出、历史文档对应。
- 最新版本目录使用中文命名，提高交付可读性。

## 不移动的内容

默认不移动以下内容：

- `backend/dist/`：PyInstaller 中间输出目录，由打包脚本自动重建，默认不作为用户交付成品归档。
- `frontend/dist/`：普通前端构建产物。
- `frontend/dist-enterprise/`：企业端前端构建产物。
- `node_modules/`、`venv/`、`backend/venv/`。
- Office 临时文件，例如 `~$...docx`。

如果用户明确要求完整保存所有中间产物，再单独扩展归档范围。

## 执行前检查

1. 确认当前仓库状态：

```powershell
git -C "." status --short
```

2. 如果存在未提交代码改动，先确认这些改动是否需要一起进入本次封装。

3. 确认当前分支和远端规则：

```powershell
git -C "." branch --show-current
git -C "." remote -v
```

约定：

- 主分支：`main`
- 远端：`AGV`
- 默认推送：`git -C "." push AGV main`

## 第一步：归档旧版本

在 `dist` 下新建：

```text
dist/旧版本/归档_YYYYMMDD_HHMM/
```

将当前 `dist` 下已有的用户交付封装目录移动进去。

默认移动匹配：

```text
dist/AGV_Dispatch_Package*
dist/AGV_Enterprise_Client*
```

不要移动：

```text
dist/旧版本
dist/最新版本
```

如果 `dist/最新版本` 中已有上一轮最新包，也将其整体移动到本次 `旧版本/归档_YYYYMMDD_HHMM/上一轮最新版本/`，避免新旧混在一起。

## 第二步：清理最新版本目录

新建或清空：

```text
dist/最新版本/
```

注意：

- 只清理 `dist/最新版本`。
- 不删除 `dist/旧版本`。
- 不删除源码、数据库、demo、docs。

## 第三步：重新封装综合版

执行：

```powershell
cmd /c build_windows_package.bat
```

脚本当前输出：

```text
dist/AGV_Dispatch_Package_v2/
```

构建成功后，将该目录移动或复制为：

```text
dist/最新版本/AGV调度系统_综合版_YYYYMMDD/
```

综合版目录应至少包含：

```text
backend.exe
start_agv.bat
data/
demo/
_internal/
```

## 第四步：重新封装企业端独立客户端

执行：

```powershell
cmd /c build_enterprise_windows_package.bat
```

脚本当前输出：

```text
dist/AGV_Enterprise_Client_v1/
```

构建成功后，将该目录移动或复制为：

```text
dist/最新版本/AGV企业端独立客户端_YYYYMMDD/
```

企业端目录应至少包含：

```text
backend.exe
start_enterprise_client.bat
data/
docs/
demo/
_internal/
```

## 第五步：封装后检查

检查最新版本目录：

```powershell
Get-ChildItem -Path "dist/最新版本" -Force
```

检查两个核心 exe：

```powershell
Get-ChildItem -Path "dist/最新版本" -Recurse -Filter "backend.exe"
```

检查两个启动脚本：

```powershell
Get-ChildItem -Path "dist/最新版本" -Recurse -Filter "start_*.bat"
```

检查 demo 资产：

```powershell
Get-ChildItem -Path "dist/最新版本" -Recurse -Filter "*.json" | Select-Object -First 20
```

## 第六步：启动验证

综合版建议验证：

```text
dist/最新版本/AGV调度系统_综合版_YYYYMMDD/start_agv.bat
```

企业端独立客户端建议验证：

```text
dist/最新版本/AGV企业端独立客户端_YYYYMMDD/start_enterprise_client.bat
```

人工观察：

- 是否能正常打开浏览器。
- 是否能进入登录页。
- 是否能登录测试账号。
- 是否能加载地图。
- 是否能打开企业设置。
- 是否能查看 demo/json 目录。

如果双击无反应，优先在 PowerShell 中直接运行 `.bat`，查看窗口输出。

## 第七步：记录本次封装

建议新增或更新：

```text
docs/release/REPACKAGE_LOG_YYYYMMDD.md
```

记录内容：

- 本次提交号。
- 打包时间。
- 综合版目录名。
- 企业端目录名。
- 验证结果。
- 已知问题。

## 第八步：提交与推送

如果本次只移动打包产物，是否提交需要看仓库策略和文件体积。

默认建议：

- 文档、脚本、配置改动：提交并推送。
- 大体积 `dist` 成品：先看是否已经被 `.gitignore` 忽略；如果未跟踪且体积很大，不强行提交。

如果需要提交：

```powershell
git -C "." status --short
git -C "." add <需要提交的文件>
git -C "." commit -m "Repackage release artifacts"
git -C "." push AGV main
```

注意：

- 不使用 `http.version=HTTP/1.1`。
- 不使用交互式 Git。
- 不随意 `amend`。
- 不使用 `reset --hard`。
- 不使用 `checkout --` 回滚用户文件。

## 后续可优化项

- 可以新增一个专门的整理脚本，例如 `tools/windows/repackage_and_organize.bat`。
- 可以让构建脚本直接输出中文目录，减少后续移动步骤。
- 可以在最新版本目录内自动生成 `版本说明.txt`。
- 可以为综合版和企业端分别生成压缩包，例如：
  - `AGV调度系统_综合版_YYYYMMDD.zip`
  - `AGV企业端独立客户端_YYYYMMDD.zip`
