# 企业端避让演示助手复测说明

## 适用范围

本说明用于复测企业端“企业避让演示助手”。该助手位于企业设置的地图方案页，用于一键载入企业拓扑避让场景：自动保存或复用地图方案、应用地图、补齐起点 AGV、导入配套任务并尝试调度。

它验证的是调度层动态避让能力，包括拓扑边占用、交汇节点预约、同廊道跟车、站点容量和等待/重规划提示。它不验证真实机器人传感器级局部避障。

## 测试前准备

1. 使用企业管理员账号登录企业端。
2. 刷新页面，确保加载的是最新前端资源。
3. 等待旧任务完成，或确认旧任务不会占用演示起点。
4. 打开企业设置，进入“地图方案”。
5. 开启“显示等待/让行标签与原因”，便于观察地图上的短标签。

## 入口位置

路径：

`企业设置 -> 地图方案 -> 企业避让演示助手`

每个演示卡片提供两个动作：

- `载入演示`：应用地图、补齐起点 AGV、导入任务并尝试调度。
- `定位起点`：在地图上高亮该演示需要的起点位置。

## 场景一：对向会车

卡片名称：`对向会车`

预期：

- 地图切换到单通道对向会车场景。
- 起点 `(1, 3)` 和 `(10, 3)` 有可调度 AGV。
- 两台车从单通道两端相向运行。
- 不应出现同格重叠或面对面穿车。
- 受阻车辆应显示“等待路段”“重规划中”或类似等待/让行标签。

异常判断：

- 两台车同时占用同一格或同一拓扑边。
- 两台车迎面直接穿过。
- 任务长期停在等待状态，另一台车也没有继续运行。

## 场景二：三车交汇

卡片名称：`三车交汇`

预期：

- 地图切换到十字交汇场景。
- 起点 `(1, 3)`、`(6, 1)`、`(10, 3)` 有可调度 AGV。
- 三台车从不同方向接近中心交汇点。
- 中心交汇点一次只允许一台车预约通过。
- 其他车辆应等待、跟车或重规划，并显示“交汇预约”“等待路段”“等待占用”等短标签。

异常判断：

- 多台车同时挤进中心交汇点。
- 等待车辆长期不恢复。
- 任务卡或 hover 中没有任何等待/重规划原因，但车辆停住不动。

## 场景三：站点出入冲突

卡片名称：`站点出入冲突`

预期：

- 地图切换到停车点、充电点和主干合流通道场景。
- 起点 `(1, 2)`、`(10, 2)`、`(1, 4)` 有可调度 AGV。
- 车辆出站、入站和经过合流点时不应同时抢占同一个入口或容量为 1 的节点。
- 停车点和充电点容量显示保持正常。
- 站点内车辆不应消失，任务完成后 AGV 应恢复可调度状态。

异常判断：

- 站点容量显示与实际车辆不一致。
- 车辆进入站点后消失。
- 合流点被多台车同时占用或长期堵死。

## 自动化检查

演示助手的配置源在前端代码中。每次修改演示地图、起点或任务后，建议运行：

```powershell
cd frontend/agv-frontend
npm run smoke:enterprise-demos
```

该检查会验证：

- 每套演示都有唯一 key、地图名和任务名。
- 每个演示起点都绑定到合法拓扑上岗节点。
- 上岗节点必须是 `station`、`parking` 或 `charge`。
- 任务起点必须覆盖演示助手声明的起点。
- 起终点不能落在障碍格外或地图外。
- 拓扑边必须引用存在的节点。

## 回归建议

- 若只改文案或样式，至少运行 `npm run smoke:enterprise-demos`、`npm run lint`、`npm run build`。
- 若改动企业端调度算法，还应运行后端 compileall、`runtime_conflict_smoke.py` 和 `runtime_long_run_smoke.py`。
- 若修改起点或拓扑节点类型，必须人工重新点击三张演示卡片，确认不再触发企业上岗点错误。
