# 动态避让设计说明

## 定位

当前系统实现的是“调度层动态避让”。它面向仓储网格和企业路网拓扑，在任务派发与车辆推进过程中判断资源占用，避免多台 AGV 在同一格点、同一拓扑节点或同一拓扑边上发生冲突。

它解决的是上层交通调度问题，不是机器人底盘的实时传感器避障问题。真实设备接入后，底层机器人仍需要自己的急停、避障和速度控制能力。

## 输入数据

- 地图尺寸：来自当前运行地图或任务 JSON 中的 `grid_cols`、`grid_rows`。
- 静态障碍：来自地图障碍物与异形有效区域，最终合并为不可通行区域。
- 拓扑节点：来自企业路网或地图方案中的 `topology.nodes`，包含节点类型、容量和坐标。
- 拓扑边：来自 `topology.edges`，包含方向、车道类型、权重和速度倍率。
- 运行态车辆：来自 AGV 当前坐标、目标坐标、`current_edge`、`current_node`、`task_id`、`motion_state`。
- 任务属性：来自任务优先级、调度模式、算法、阶段路径和绑定车辆。

## 主要机制

- 静态避障：`astar` 在规划阶段把障碍格和无效格作为不可通行区域。
- 节点占用：拓扑节点容量默认为 1，进入节点前会检查其他 AGV 是否正在占用或即将占用。
- 边占用：AGV 在拓扑边上运行时会记录 `current_edge` 和进度，其他车辆进入同一边或反向边时需要等待或重试。
- 同路不超车：同一走廊内，如果前车占用目标方向，后车不能直接穿越前车。
- 对向相遇：双向边或窄通道上出现对向车辆时，系统按优先级、任务顺序和 AGV 编号决定等待关系。
- 重规划倾向：`astar` 会根据运行态占用、边冲突和优先级增加代价，优先寻找更稳的路线。
- 死锁恢复：长时间等待后会触发重试或恢复逻辑，避免任务永久卡在占用状态。

## 调度流程

1. 任务进入队列，携带 `dispatch_mode`、`dispatch_algorithm` 和优先级。
2. 调度服务选择任务与 AGV。自动模式按优先级、距离和可达性匹配，手动模式优先绑定指定 AGV。
3. 路径规划读取地图障碍、有效区域、拓扑和运行态占用。
4. AGV 推进时更新节点、边、进度、速度倍率和运动状态。
5. 如果前方资源被占用，车辆进入等待或重试状态。
6. 资源释放后，车辆继续推进；若长期无法推进，则触发恢复或重新调度。

## 代码位置

- `backend/app/services/schedule_service.py`：任务与 AGV 匹配、算法选择、派发与恢复入口。
- `backend/app/utils/path_planner.py`：`simple`、`astar`、拓扑边代价、运行态占用和动态阻塞处理。
- `backend/app/utils/agv_movement.py`：AGV 时间步推进、边占用、节点占用、等待、让行和死锁恢复。
- `backend/app/utils/warehouse_map.py`：地图尺寸、障碍格、有效区域、拓扑读取与归一化。
- `backend/app/models/agv.py`：AGV 运行态字段，例如 `current_edge`、`edge_progress`、`motion_state`。

## 当前边界

- 当前避让以离散格点和拓扑资源为粒度，不做连续空间碰撞检测。
- 当前速度控制是展示和调度意义上的速度倍率，不等同于真实底盘控制器。
- 当前没有接入真实传感器，因此无法处理突然闯入的人、叉车或未建模障碍物。
- 当前站点合并主要是视觉和容量语义，还没有升级为完整的站级调度实体。

## 下一步增强

- 前端可视化等待、让行、重规划状态，让用户更容易看懂“为什么停下”。
- 引入走廊分组、等待位和缓冲区，让窄通道交通规则更明确。
- 增强拓扑合法性校验，保存前提示孤立节点、重复边、单向断路和容量异常。
- 如果接入真实设备，再叠加传感器级局部避障、连续速度规划和设备通信协议。
