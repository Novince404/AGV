# 答辩通过后产品化收口计划 2026-04-22

## 1. 当前结论

毕业设计答辩已经通过，系统已达到“可演示、可讲解、可交付答辩材料”的阶段目标。

从现在开始，项目主线从“答辩可用版”切换为“产品化收口版”。后续不再围绕答辩材料堆功能，而是优先做三件事：

- 把第四阶段 4A-4F 的状态从旧的“待答辩/待人工签收”改成清晰的产品化状态。
- 把动态避让、站级调度实体、企业独立客户端等高价值方向拆成可控迭代。
- 保持现有稳定功能，不为了增强而重做已经可用的个人端、企业端、地图和任务链路。

## 2. 4A-4F 答辩后状态

### 4A 企业角色工作台与反馈基础

状态：`答辩版完成，产品化维持`

已完成：

- 企业管理员、企业操作工、企业后勤岗分角色工作台。
- 企业内部请求 / 报错反馈。
- 平台 Bug 反馈。
- 平台管理员治理优先工作台。
- 小铃铛通知入口与反馈状态链路。

后续只做低风险优化，例如通知未读状态细节、列表筛选、批量处理和更清晰的反馈分类。

### 4B 企业拓扑与地图/路线底层模型

状态：`核心完成，产品化继续增强`

已完成：

- 企业路网拓扑编辑、节点、连边、车道类型、方向、速度倍率。
- 拓扑保存与运行态读取。
- 主干道接入 / 离开、拓扑边速度倍率和普通网格回退。
- 常用点位与拓扑节点语义分离。

后续重点：

- 拓扑保存后的可视化确认和变更摘要。
- 拓扑版本管理或一键恢复。
- 更明确的路网合法性校验，例如孤立节点、单向断路、重复边、容量异常。

### 4C 连续移动与时间步推进

状态：`答辩版完成，产品化维持`

已完成：

- AGV 运行状态从简单跳格升级为带路径推进和运动状态的展示。
- 派发后路径立即显示。
- 任务完成后路径、起点、终点及时收口。
- 地图状态、任务状态、车辆状态联动显示。

后续只做观感类优化，例如轨迹回放、速度曲线、运行耗时统计。

### 4D 冲突预占、双车道与优先级避让

状态：`已有调度层动态避让，产品化重点增强`

已完成：

- 拓扑边运行时预占。
- 节点 / 边占用检测。
- 同路不超车。
- 双向相遇等待 / 让行。
- 基于任务优先级、绕行代价和稳定排序的重规划倾向。
- 死锁超时后的重试和恢复。

当前边界：

- 这是调度层动态避让，不是传感器级动态避障。
- 当前避让主要基于离散网格、拓扑边、节点容量和任务优先级。
- 还没有引入连续空间速度规划、真实传感器感知、DWA / ORCA / MPC 等机器人局部避障算法。

下一步建议：

- 增加一个稳定的“动态避让演示场景”JSON，让多车在窄通道或交汇口触发等待、让行、恢复。
- 在前端更清楚显示“等待前方占用 / 正在让行 / 正在重规划”。
- 增加一份动态避让说明文档，明确已有能力和未来可扩展方向。

### 4E 回仓、回充与空闲策略

状态：`核心完成，产品化继续增强`

已完成：

- 空闲回仓。
- 低电量回充。
- 充电点 / 停车点容量。
- 充电等待状态。
- 充满释放阈值和主动充电阈值。
- 回仓 / 回充卡死恢复。

后续重点：

- 站内车辆列表、`n/m`、悬停信息继续做一致性复核。
- 充电策略可以扩展为更完整的电池健康模型，例如低电阈值、最佳充电窗口、任务前电量预估。
- 回仓策略可以增加“最近空闲站 / 指定站 / 均衡站内容量”三种策略。

### 4F 企业独立客户端

状态：`答辩演示可用，产品化验收继续`

已完成：

- 企业独立客户端打包链。
- `start_enterprise_client.bat` 启动入口。
- 包内后端 smoke。
- 三角色登录 API 链路。
- `fail to fetch` 关键问题修复。

后续重点：

- 重新做一次完整浏览器人工验收。
- 明确企业独立客户端版本号、数据文件位置、日志位置和故障排查方式。
- 把“演示包”与“正式企业客户端包”的边界写清楚。

## 3. 后续优先级

### P0 保持稳定

- 不重做个人端和企业端主界面。
- 不在短期内重构数据库表结构。
- 不把视觉聚合站点直接升级为真实站级实体，除非先完成单独设计。

### P1 产品化收口

- 更新第四阶段验收文档，把“答辩通过”与“产品化待增强”分开。
- 补一个动态避让演示场景。
- 补动态避让设计说明。
- 对企业独立客户端做一次完整人工验收记录。

### P2 系统设计增强

- 动态避让可视化：等待、让行、重规划状态更清楚。
- 拓扑合法性校验：保存前发现断路、重复边、容量异常。
- 站点详情增强：站内车辆列表、泊位状态、入口/出口提示更稳定。

### P3 中长期扩展

- 站级调度实体：停车站 / 充电站从视觉聚合升级为真正调度对象。
- 更强交通控制：等待位、缓冲区、走廊分组、主干道优先级策略。
- 真实设备通信：WebSocket、MQTT、串口或 PLC 对接。
- 传感器级动态避障：连续空间避障和速度规划。

## 4. 动态避让定位

当前系统已经具备“调度层动态避让”：

- 根据 AGV 当前占用的节点、边和目标点判断冲突。
- 根据优先级、绕行代价、任务创建顺序和 AGV 编号决定谁等待、谁重规划。
- 遇到长期等待时触发重试或死锁打破。

答辩或后续系统说明中建议这样表述：

> 当前系统实现的是面向仓储网格和企业路网拓扑的调度层动态避让。它能处理同路不超车、对向相遇、节点占用、站点容量和部分死锁恢复。下一步如果接入真实机器人，可以在此基础上增加传感器级局部避障和连续速度规划。

## 5. 下一步建议执行顺序

1. 新增动态避让演示 JSON，覆盖窄通道、交汇口、等待恢复。
2. 新增动态避让设计说明，写清楚调度层避让和传感器级避障的边界。
3. 更新企业独立客户端人工验收记录。
4. 评估是否进入“站级调度实体”专题设计。
5. 再决定是否做真实设备通信或更完整产品打包。
